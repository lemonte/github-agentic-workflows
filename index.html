🐛 Testando o Workflow de Issue Clarifier

Este repositório possui um workflow agêntico que monitora novas issues abertas. Quando uma issue é criada com descrição vaga ou incompleta, o Copilot automaticamente pede mais detalhes ao autor.

🧪 Como testar: Abra uma issue mal descrita neste repositório — por exemplo, apenas com o título "bug" ou "não está funcionando" — e observe o Copilot responder automaticamente pedindo mais informações.

📋 Passo a passo para o teste

  1. Acesse a aba Issues deste repositório no GitHub.

  2. Clique em "New issue" para criar uma nova issue.

  3. Preencha apenas o título com algo vago — por exemplo: "problema", "erro" ou "tá quebrando". Deixe o campo de descrição vazio ou com poucas palavras.

  4. Clique em "Submit new issue" para criar.

  5. Aguarde alguns instantes. O GitHub Copilot irá analisar a issue e postar automaticamente um comentário solicitando mais detalhes sobre o problema.

💡 O que o workflow faz?

O arquivo .github/workflows/issue-clarifier.md instrui o Copilot a:

  • Analisar o conteúdo da issue assim que ela é aberta
  • Identificar se a descrição está incompleta ou pouco detalhada
  • Postar um comentário amigável pedindo informações adicionais ao autor
  • Ajudar a manter o backlog organizado e acionável

🏷️ Gatilho do Workflow

Este workflow é disparado automaticamente em:

  • issues: opened — toda vez que uma nova issue é criada

🔀 Workflow: Fluxograma de Alterações de Pull Request

Ao abrir um Pull Request, o Copilot analisa automaticamente os arquivos modificados e gera um fluxograma de regras de negócio diretamente como comentário no PR — destacando visualmente o que foi removido e o que foi adicionado.

📋 Como o Workflow age ao abrir um PR

  1. O Copilot é acionado assim que um novo Pull Request é aberto.

  2. Ele lê todos os arquivos modificados no PR e identifica as regras de negócio impactadas.

  3. Gera um fluxograma Mermaid mostrando o antes e o depois do fluxo de negócio.

  4. Posta o fluxograma como comentário no PR, acompanhado de um resumo em linguagem natural das mudanças de negócio.

  5. Se as alterações forem apenas técnicas (refatoração, lint, tipagem), o Copilot informa isso explicitamente, sem gerar fluxograma.

🏷️ Gatilho do Workflow

Este workflow é disparado automaticamente em:

  • pull_request: opened — toda vez que um novo PR é aberto

Arquivo de configuração: .github/workflows/pr-change-flowchart.md

⚙️ Como os Workflows Agênticos Funcionam

Os workflows agênticos do GitHub Copilot utilizam arquivos .md dentro da pasta .github/workflows/ para definir o comportamento do agente. Diferente dos workflows tradicionais do GitHub Actions (arquivos .yml), estes arquivos contêm instruções em linguagem natural para o Copilot.

📁 Estrutura de um Arquivo de Workflow Agêntico

Cada arquivo .md de workflow possui:

  • Frontmatter YAML — define o gatilho (on:), permissões e outputs seguros
  • Corpo em Markdown — instrução em linguagem natural para o Copilot executar

Exemplo de frontmatter:

---
on:
  issues:
    types: [opened]

permissions: read-all

safe-outputs:
  add-comment:
---

🤖 Workflows deste Repositório

  • issue-clarifier.md issues: opened

    Analisa novas issues e solicita mais detalhes quando a descrição está incompleta.

  • pr-change-flowchart.md pull_request: opened

    Gera automaticamente um fluxograma de regras de negócio ao abrir um Pull Request, destacando em vermelho o que foi removido e em verde o que foi adicionado.

🔐 Segurança e Permissões

Os workflows agênticos operam com o princípio de menor privilégio. O campo safe-outputs limita as ações que o Copilot pode executar — neste caso, apenas postar comentários (add-comment) — sem acesso a escrita no código, configurações do repositório ou outros recursos sensíveis.
